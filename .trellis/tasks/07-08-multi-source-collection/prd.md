# 多源采集与数据源切换（父任务）

## 背景

系统当前只有单一采集源 `creprice.cn`（城市→区县 月度 ¥/㎡ 均价 + 价格区间分布）。该源近期出现两类问题：

1. **IP 级限流**：数据中心 IP 高频访问触发 SSL EOF / 断连（见 `07-07-full-data-crawl` notes）。
2. **源站自身不稳定**：用户反馈疑似源站出现问题（非纯粹境外 IP 不可达）。

单源导致系统数据可用性完全依赖一个不可控的第三方站点，风险高。需要引入**多个可靠采集源**，实现多源混合与容错，并让运营者能在前端**切换/选择数据源**。

## 目标

1. 新增 ≥2 个可靠采集源，降低对 creprice 单源的依赖。来源优先级：
   - **政府/官方数据**（最可靠）：国家统计局房价指数、商品房销售面积/额等。
   - **公开数据集**：Kaggle 等平台的中国城市房价历史数据集（用于历史回填）。
   - **中介站**（形状最匹配但反爬重）：链家/贝壳/安居客 城市/商圈 ¥/㎡ 均价——按可行性研究结果决定是否实现。
2. 采集框架支持**异构数据源**：不同源能力不同（有无区县、有无分布、指数 vs 绝对均价），编排层按源能力自适应，不再硬编码 creprice 的流程假设。
3. 前端管理端支持**数据源选择/切换**：可查看可用源、设置当前采集源、按源触发采集。
4. 保持 creprice 现有全量数据与流程**不回归**。

## 约束

- **代理**：可用 `http://Default:***@10.0.0.1:2260`，但实测出口为美国数据中心 IP，对政府站/部分站点连通性反而更差。代理仅作为可选手段，不作为解决境外访问的银弹。数据源选择应优先"境外可达 + 无强反爬"的源。
- **反爬红线**：延续现有严格串行 + 限速策略；触发封禁应退避而非提速。禁止为绕过风控做违法或破坏性操作；验证码/滑块仅在有合规开源方案时尝试。
- **全容器化**：后端/前端在 docker compose 内运行，勿在宿主机跑 uvicorn/vite（见项目 memory）。
- **数据模型兼容**：改动 `price_snapshot` 等核心表需 alembic 迁移且不破坏既有查询/前端。

## 子任务拆分（随研究结论确定，父任务负责集成验收）

- **child A — 多源框架泛化 + 数据源切换**（后端抽象/编排/API + 前端切换 UI）：解耦硬编码 `SOURCE_NAME`，加源能力声明，加"可用源列表/当前源"API 与前端卡片。这是其余源接入的地基。
- **child B — 国家统计局数据源接入**（政府可靠源）。
- **child C — Kaggle/公开数据集导入**（历史回填，文件/API 导入）。
- **child D — 中介站源接入**（按 `agency-antibot.md` 研究结论决定实现范围，可能降级为"最小可行 + 明确标注不稳定"或暂缓）。

> 拆分与优先级在研究（`research/*.md`）完成、design.md 定稿后落定；child D 视可行性可能收缩或延后。
